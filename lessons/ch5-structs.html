Bloc 3 · Chapitre 5

Structs & Méthodes

Brown Book Ch.5 · Référence std

1. Pourquoi les structs ?

En C tu regrouperais une lecture BME280 dans une struct : 

// C
typedef struct {
    int32_t  temperature;  // centièmes de degrés
    uint32_t humidity;
    uint32_t pressure;
    bool     valid;
} SensorReading;

Rust fait la même chose, mais la struct fait partie du système de types — pas juste un bloc mémoire nommé.

2. Syntaxe de base

// Déclaration
struct SensorReading {
    temperature_raw: i32,
    humidity_raw:    u32,
    pressure_raw:    u32,
    valid:           bool,
}

// Instanciation
let reading = SensorReading {
    temperature_raw: 2534,
    humidity_raw:    6421,
    pressure_raw:    101325,
    valid:           true,
};

Field init shorthand

Quand la variable locale a le même nom que le champ, on peut omettre la clé :

fn new(temperature_raw: i32, humidity_raw: u32, pressure_raw: u32) -> SensorReading {
    SensorReading {
        temperature_raw,   // shorthand : identique à temperature_raw: temperature_raw
        humidity_raw,
        pressure_raw,
        valid: true,
    }
}

Struct update syntax

Pour créer un nouvel objet en changeant seulement quelques champs :

let base = SensorReading { temperature_raw: 2500, humidity_raw: 6000, pressure_raw: 101325, valid: true };
let updated = SensorReading { pressure_raw: 98000, ..base };
⚠ Move ou Copy ?
Le .. move les champs non-Copy hors de base (ex: String). Si tous les champs sont Copy (i32, u32, bool…), base reste utilisable. La règle est la même que pour n'importe quelle valeur — le borrow checker s'applique champ par champ.

3. Autres formes de structs

Tuple structs

Quand les noms de champs n'apportent rien, on numérote avec .0, .1, etc. :

struct RgbColor(u8, u8, u8);

let red = RgbColor(255, 0, 0);
println!("{}", red.0);  // 255

Utilité embedded : encoder des paires (adresse I2C, registre), des vecteurs 3D en virgule fixe, etc.

Unit-like structs

Sans aucun champ — utile pour implémenter des traits sans état (ex: marqueur de type) :

struct Uninitialized;   // type-state : on reviendra là-dessus en Bloc 4

4. Ownership & durée de vie dans les champs

Pourquoi utiliser String plutôt que &str dans un struct ? Parce qu'un champ &str implique que les données vivent ailleurs, et le compilateur a besoin d'une annotation de durée de vie pour le savoir — concept du Bloc 4. 

// ✅ La struct possède ses données — pas de problème de durée de vie
struct DeviceName {
    label: String,
}

// ❌ Refus du compilateur sans annotation de lifetime
struct DeviceName {
    label: &str,   // error[E0106]: missing lifetime specifier
}
🔌 Angle embedded
Dans du code no_std, on préfère souvent éviter String (qui alloue). On utilise &'static str (chaîne en flash) ou un tableau [u8; N]. On y reviendra quand on abordera les lifetimes et heapless.

5. Méthodes — le bloc impl

Les méthodes sont des fonctions associées à un type, déclarées dans un bloc impl. Le premier paramètre est toujours une forme de self :

Signature Ce que ça fait Analogie C
&self Lecture seule const SensorReading*
&mut self Mutation SensorReading*
self Consommation (move) Pas d'équivalent direct
Pas de self Fonction associée (constructeur, factory) sensor_reading_init() 
impl SensorReading {
    // Fonction associée — appelée avec SensorReading::new(…)
    pub fn new(temperature_raw: i32, humidity_raw: u32, pressure_raw: u32) -> Self {
        let valid = temperature_raw >= -4000
            && temperature_raw <= 8500
            && humidity_raw <= 10000
            && pressure_raw > 0;
        Self { temperature_raw, humidity_raw, pressure_raw, valid }
    }

    // Méthode lecture — emprunt partagé
    pub fn temperature_celsius(&self) -> f32 {
        self.temperature_raw as f32 / 100.0
    }

    // Méthode mutation — emprunt exclusif
    pub fn invalidate(&mut self) {
        self.valid = false;
    }
}
✅ Essaie dans le playground
play.rust-lang.org — colle le struct + impl ci-dessus, puis appelle SensorReading::new(2534, 6421, 101325) dans main(). Essaie ensuite d'appeler invalidate() sans déclarer la variable mut — lis l'erreur.

6. #[derive(Debug)]

Rust refuse d'afficher un struct avec {:?} à moins qu'il implémente le trait Debug. La façon la plus simple : laisser le compilateur le dériver automatiquement. 

#[derive(Debug)]
struct SensorReading { /* … */ }

let r = SensorReading::new(2534, 6421, 101325);
println!("{r:?}");   // SensorReading { temperature_raw: 2534, … }
println!("{r:#?}");  // version pretty-printed
Info
#[derive(Debug)] est une macro procédurale — le compilateur génère le code d'affichage à ta place. On reviendra sur les macros en Bloc 3. Pour l'instant retiens : sans Debug, {:?} est une erreur de compilation.

7. Méthodes vs fonctions libres — quand choisir ?

Rust n'impose pas les méthodes. Une fonction fn area(r: &Rectangle) -> f64 et une méthode fn area(&self) -> f64 font la même chose. Mais les méthodes gagnent sur plusieurs points :

  • Autocomplétion (r. dans ton éditeur)
  • Namespace clair (SensorReading::new vs new_sensor_reading)
  • Implémentation de traits (on ne peut pas implémenter un trait avec des fonctions libres)
🔌 Pattern type-state (aperçu Bloc 4)
En embedded, on utilise souvent les fonctions associées comme constructeurs d'état : Sensor::<Uninitialized>::new() retourne un type différent de Sensor::<Running>::init(). Le compilateur rend alors impossible d'appeler read() sur un capteur non initialisé. C'est ce qu'on appellera le type-state pattern — Bloc 4.
