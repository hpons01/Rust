📦 Récapitulatif S5 — Ownership : move, clone, drop

Stack vs Heap

  • La stack stocke des données de taille fixe, allouées/désallouées automatiquement à chaque appel de fonction.
  • Le heap stocke des données de taille variable (String, Vec, Box…). Rust en gère la libération via l'ownership.

Les 3 règles d'Ownership

1. Chaque valeur a exactement un seul owner.
2. Il ne peut y avoir qu'un owner à la fois.
3. Quand l'owner sort de portée, la valeur est droptée (mémoire libérée).

Move (déplacement)

  • Pour les types heap (String, Vec…), let s2 = s1 transfère l'ownership à s2.
  • s1 devient invalide — le compilateur refuse de l'utiliser.
  • Objectif : éviter le double-free.
let s1 = String::from("hello");
let s2 = s1;                  // move : s1 invalide
// println!("{}", s1);       // ❌ erreur de compilation
println!("{}", s2);          // ✅ ok

Clone (copie profonde)

  • .clone() crée une copie indépendante des données heap.
  • Explicite (coûteux) — Rust ne le fait jamais en silence.
let s1 = String::from("hello");
let s2 = s1.clone();         // copie profonde
println!("{} {}", s1, s2);   // ✅ les deux valides

Copy trait

  • Les types simples (i32, bool, f64, char, tuples de Copy…) implémentent Copy.
  • Ils sont copiés au lieu d'être movés — l'original reste valide.
  • String, Vec, Box n'implémentent PAS Copy.

Drop

  • Quand une variable sort de portée (}), Rust appelle automatiquement drop().
  • Libère la mémoire heap sans appel manuel à free().
🔗 Récapitulatif S6 — References, borrowing, &mut

Principe fondamental

Pointer Safety Principle : une donnée ne peut jamais être aliasée ET mutée en même temps.

Références immutables — &T

  • Emprunte une valeur en lecture seule sans en prendre l'ownership.
  • Plusieurs &T simultanés : autorisé.
  • Le borrow checker retire les permissions W+O à la variable empruntée pendant la durée de l'emprunt.

Références mutables — &mut T

  • Emprunte avec permission d'écriture.
  • Une seule &mut T à la fois — aucun autre accès simultané.

Les 2 règles des références

1. Soit N références immutables (&T), soit exactement une référence mutable (&mut T) — jamais les deux en même temps.
2. Les références doivent toujours pointer vers des données valides (pas de dangling reference).

Dangling references

  • Rust interdit de retourner une référence vers une variable locale (qui sera droptée à la fin de la fonction).
  • Détecté à la compilation — zéro overhead runtime.
fn dangle() -> &String {   // ❌ ne compile pas
    let s = String::from("hi");
    &s  // s droppé ici → référence invalide
}

NLL — Non-Lexical Lifetimes

  • Depuis Rust 2018 : la durée de vie d'une référence se termine à son dernier point d'utilisation, pas à la fin du bloc.
  • Permet de réutiliser une variable après la fin effective d'un emprunt.
let mut v = vec![1, 2, 3];
let r = &v[0];
println!("{}", r);    // ← dernière utilisation de r (NLL)
v.push(4);           // ✅ ok : r n'est plus actif

Analogie embarquée

  • Pensez à un registre hardware : soit plusieurs drivers le lisent, soit un seul l'écrit. Les deux simultanément = undefined behavior.
