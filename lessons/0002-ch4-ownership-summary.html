Ch.4 — Ownership, Références & Slices

Bloc 2 · Leçon de synthèse · 2026-06-14

1 — Le modèle mémoire : Stack vs Heap

Avant les règles d'ownership, il faut avoir en tête où les données vivent. Tu connais déjà la distinction depuis C — Rust l'exploite explicitement dans ses règles.

  STACK                          HEAP
  ┌─────────────────────┐        ┌─────────────────────────────┐
  │  frame main()       │        │                             │
  │  ┌───────────────┐  │  ptr ──►  H  e  l  l  o  !         │
  │  │ s             │  │        │                             │
  │  │  ptr  ──────────────────► └─────────────────────────────┘
  │  │  len = 6      │  │
  │  │  cap = 6      │  │
  │  └───────────────┘  │
  │                     │
  │  x: i32 = 42        │        ← vit entièrement sur la stack
  └─────────────────────┘
Stack
taille connue à compile-time
Heap
alloué à runtime, taille variable
String (vs &str)
pointeur stack → données heap

Le pointeur fat d'un String est un triple (ptr, len, cap) sur la stack. Les octets eux sont sur le heap — c'est pour ça que String a des règles d'ownership et que i32 n'en a pas besoin.

2 — Les trois règles d'Ownership

Règle 1. Chaque valeur a un seul owner.
Règle 2. Il ne peut y avoir qu'un seul owner à la fois.
Règle 3. Quand l'owner sort de scope, la valeur est droppée (libérée).

Pas de free() explicite. Pas de GC. Le compilateur insère le drop automatiquement — et garantit qu'il n'y a qu'un seul drop. Fin du double-free en C.

Move vs Copy

  ┌── MOVE (types heap : String, Vec…) ───────────────────────────────┐
  │                                                                     │
  │   let s1 = String::from("hello");                                  │
  │   let s2 = s1;   ←── ownership TRANSFÉRÉ à s2                     │
  │                                                                     │
  │   AVANT       s1 ──► [ptr|len|cap]  →  heap: "hello"              │
  │   APRÈS       s1  ✗ (invalidé)                                     │
  │               s2 ──► [ptr|len|cap]  →  heap: "hello"              │
  │                                                                     │
  │   Bénéfice : pas de double-free, pas de copie heap silencieuse     │
  └─────────────────────────────────────────────────────────────────────┘

  ┌── COPY (types stack : i32, bool, f64, char, tuples…) ─────────────┐
  │                                                                     │
  │   let x: i32 = 42;                                                  │
  │   let y = x;   ←── COPIE bitwise sur la stack, x reste valide      │
  │                                                                     │
  │   AVANT/APRÈS  x = 42    y = 42   (indépendants)                   │
  │                                                                     │
  │   Possible car taille fixe + pas d'indirection heap                 │
  └─────────────────────────────────────────────────────────────────────┘
Règle de décision rapide : si le type implémente le trait Copy, l'assignation copie. Sinon, elle move. Corollaire : un type avec un Drop custom ne peut pas implémenter Copy.

3 — Références & Borrowing

Donner l'ownership pour lire une valeur, puis la récupérer, c'est lourd. Les références permettent d'emprunter une valeur sans en prendre l'ownership.

  let s = String::from("hello");
  let r = &s;              ←── référence immuable

  Mémoire :
  ┌──────────┐        ┌────────────────────┐       ┌──────────────┐
  │  r       │  ptr   │  s                 │  ptr  │  heap        │
  │  ────────────────►│  ptr ─────────────────────►│  "hello"     │
  └──────────┘        │  len = 5           │       └──────────────┘
                      │  cap = 5           │
                      └────────────────────┘

  r emprunte s — s reste owner, r n'a pas le droit de dropper.

Les deux règles du Borrow Checker

À tout moment, tu peux avoir soit :
  • N références immuables (&T), ou
  • 1 référence mutable (&mut T) — pas les deux en même temps. 
  READERS (immuables)          WRITER (mutable)
  ┌────────────────────────┐   ┌────────────────────────┐
  │  &s   │  &s   │  &s  │   │       &mut s            │
  │  r1      r2      r3   │   │  (accès exclusif)       │
  │   OK — lecture partagée│   │  Aucun autre borrow    │
  └────────────────────────┘   └────────────────────────┘
       MAIS pas de &mut s            MAIS pas de &s

  Analogie registre embarqué : plusieurs ISR peuvent lire un registre
  en même temps, mais une seule peut écrire (sinon race condition).

NLL — Non-Lexical Lifetimes

Depuis Rust 2018, le borrow checker comprend que la durée de vie d'un borrow se termine à la dernière utilisation, pas à la fermeture de l'accolade.

let mut v = vec![1, 2, 3];
let r = &v[0];          // borrow immuable commence ici
println!("{}", r);     // <── DERNIÈRE utilisation de r : borrow se termine ici
v.push(4);             // OK — plus aucun borrow actif

Référence pendante = erreur à compile-time

fn dangle() -> &String { // ← le compilateur refuse
    let s = String::from("hello");
    &s                         // s est droppé à la fin, &s serait une dangling ref
}                              // erreur : `s` does not live long enough

En C, c'est un UB silencieux. En Rust, c'est une erreur de compilation. L'invariant : une référence ne peut pas vivre plus longtemps que la valeur qu'elle emprunte.

4 — Le modèle de permissions (Brown Book)

Le Brown Book formalise les droits du borrow checker avec trois permissions :

PermissionSignifieQui l'a
R Read Lire la valeur Owner, &T, &mut T
W Write Modifier la valeur Owner mut, &mut T
O Own Dropper ou move Owner uniquement 
  let mut s = String::from("hi");   // s : R W O
  let r1 = &s;                      // s : R . .  (W et O suspendus)
                                    // r1: R . .
  let r2 = &mut s;                  // s : . . .  (tout suspendu)
                                    // r2: R W .
  // r2 doit disparaître avant qu'on puisse reborrow s immuablement

5 — Slices : &[T] et &str

Une slice est une référence à une sous-séquence contiguë d'une collection. Elle est un fat pointer : (ptr, len).

  let s = String::from("hello world");
  let hello = &s[0..5];   ←── &str : slice de String

  Mémoire :
  ┌──────────────┐        ┌───────────────────────────────┐
  │  s           │  ptr   │  h  e  l  l  o     w  o  r  l  d  │
  │  ptr ───────────────► │                               │
  │  len=11      │        │  └─────────┘                  │
  │  cap=11      │        └───────────────────────────────┘
  └──────────────┘             ▲
  ┌──────────────┐             │
  │  hello       │   ptr ──────┘
  │  ptr         │   len = 5
  │  len = 5     │
  └──────────────┘

  hello est un &str : pas de capacity, pas d'ownership.

Pourquoi &str plutôt que &String en paramètre ?

// Moins flexible — accepte seulement &String
fn first_word(s: &String) -> &str { … }

// Idiomatic — accepte &String ET &str (string literals)
fn first_word(s: &str) -> &str { … }

Un &String se coerce automatiquement en &str, donc accepter &str est plus général. C'est le pattern idiomatique.

Slices sur Vec : &[T]

let v = vec![1u8, 2, 3, 4, 5];
let mid: &[u8] = &v[1..4];   // [2, 3, 4] — fat pointer (ptr, len=3)

En embarqué, les slices sont omniprésentes : buffers DMA, frames SPI/UART. Une fonction qui accepte &[u8] fonctionne sur un tableau statique, un Vec, ou la moitié d'un buffer — même API.
